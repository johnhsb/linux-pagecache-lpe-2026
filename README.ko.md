# linux-pagecache-lpe-2026

**Copy Fail · Dirty Frag 계열** — 2026년 Linux 커널 페이지 캐시 로컬 권한상승(LPE) 취약점 8종(그중 5종은 자동 완화 지원)에 대한 **임시 완화 스크립트**와 대응 가이드.

[🇺🇸 English version](README.en.md)

---

## 핵심 요약

- **대상은 총 8개 CVE지만 스크립트는 6개만 자동화한다.** 전부 "파일 기반 페이지 캐시를 정상 쓰기 대상처럼 취급해 제자리(in-place) 쓰기를 수행"하는 동일 실패 유형을 공유하지만, 트리거 경로가 달라 **패치는 CVE마다 별도**. [mitigate-cve-2026.sh](mitigate-cve-2026.sh)는 31431/43284/43500/46300/43503/46331 6건을 자동 완화하며, 추가로 확인된 43494(PinTheft)/31694(FUSE)는 가이드에 수동 절차만 정리했다.
- **실제 위험도는 3단계로 갈린다.**
  - 🔴 **1순위**: Copy Fail(31431) — 계열 통칭은 "Dirty Frag"지만, 8건 중 **유일하게 CISA KEV에 등재**(2026-05-01)되었고 **실제 공격 정황이 확인**됐으며, `AF_ALG` 소켓은 비특권 네임스페이스 없이 모든 로컬 계정이 열 수 있어 **트리거 조건이 가장 낮다**. 부분 적용만 가능하다면 `algif_aead` 차단부터 할 것.
  - 🟠 **2순위**: Dirty Frag(43284/43500)·Fragnesia(46300)·DirtyClone(43503)·pedit COW(46331) — 전부 비특권 user/net 네임스페이스로 CAP_NET_ADMIN을 얻어야 도달 가능한 동일 전제조건, 공개 PoC와 벤더 RHSB 보유.
  - ⚪ **기타**: PinTheft(43494)·FUSE 캐시 오버플로(31694) — 단, "기타=위험 낮음"이 아니다. 이유는 CVE마다 다르므로 대상 CVE 표와 가이드 5.6절 참고.
- **왜 완화 스크립트가 필요한가**: 궁극적 해결책은 커널 패치지만, 상용 운영 장비는 긴급 재기동 승인 지연, 벤더사의 특정 커널 고정, 패치 미 출시, 변경관리 의사결정 지연으로 즉시 패치가 어려운 경우가 흔하다. 이 스크립트는 그 공백을 메우는 **압박 완화책(compensating control)**이다.
- **조치 방식**: 취약 코드로 가는 커널 모듈(`esp4`/`esp6`/`rxrpc`/`algif_aead`)의 자동 로드를 차단. **Red Hat이 RHSB-2026-003에서 직접 제시한 완화 절차와 동일**하며, 변경관리 승인 근거로 제시 가능. 재부팅 불필요, `--rollback`으로 즉시 원복.
- **완전한 해결책은 패치뿐**: 특히 DirtyClone(43503)의 근본 결함은 커널 코어(`__pskb_copy_fclone()`)에 있어 모듈 차단만으로는 부분 방어에 그친다. 완전 보호는 DirtyFrag+Fragnesia 체인 전체(업스트림 v7.1-rc5 이상)가 적용된 커널에서만 가능하다.

```bash
sudo bash mitigate-cve-2026.sh --dry-run    # 1) 변경 없이 점검만
sudo bash mitigate-cve-2026.sh              # 2) 대화형 적용
sudo bash mitigate-cve-2026.sh --rollback   # 3) 필요 시 원복
```

> ⚠️ 반드시 **root**로 실행. 비root는 사용 신호 탐지가 조용히 실패해 오판 위험이 있어 스크립트가 실행을 거부한다.

## 확인 필요 (적용 전 재검증할 것)

- **CISA KEV 등재/실공격 정황은 시간에 따라 바뀐다.** 적용 시점에 [CISA KEV 카탈로그](https://www.cisa.gov/known-exploited-vulnerabilities-catalog)에서 8개 CVE를 재조회할 것.
- **Oracle Linux 9(UEK)의 CVE-2026-43503 어드바이저리는 2026-07 기준 미발행.** 최신 UEK 커널이라도 해당 CVE가 패치되지 않았을 수 있으니 최신 상태를 다시 확인할 것.
- **"RHEL 계열이니 미해당"이라는 일반화는 배포판마다 검증 없이는 성립하지 않는다.** PinTheft(43494)는 정식 RHEL은 미해당이지만 **Oracle Linux(UEK)는 미해당이 아니다** — RDS는 Oracle RAC 인터커넥트용 프로토콜이라 Oracle Linux에는 실제로 `CONFIG_RDS=m`이 확인됐다(2026-07-23 직접 검증, 가이드 2.7절). 커널 6.16 이상을 직접 추적하는 시스템이라면 FUSE(31694) 쪽도 우선순위를 올려 재평가할 것.
- 문서 내 CVE 정보는 **2026-07-22에 OSV/NVD/RHSB 원본으로 재검증**한 스냅샷이다. 벤더 어드바이저리·패키지 버전은 계속 바뀌므로 적용 전 배포판 최신 정보로 재확인할 것.

## 특징

- **배포판 패치 자동 확인** — changelog에서 CVE ID를 찾아 이미 패치된 것은 조치 대상에서 제외. 스크립트 지원 대상 5개 전부 해결됐으면 변경 없이 종료.
- **벤더 어드바이저리 조회(`--online`)** — changelog에 CVE ID가 없는 커널(Oracle UEK 등)에서 "수정본 배포됐으나 미설치"인 CVE를 취약 확정.
- **fail-safe 기본값** — 패치 여부를 증명할 수 없으면 조치 대상으로 남김. 조치를 건너뛰는 판정은 1차 출처로 확인된 것 하나(RxRPC/RHEL)뿐.
- **빌트인(`=y`) 감지** — 모듈 차단이 통하지 않는 경우를 구분해 경고하고 대안을 안내.
- **"미사용"과 "확인불가"를 구분** — 탐지 실패를 신호 없음으로 뭉뚱그리지 않아, IPsec 사용 서버를 미사용으로 오판해 VPN을 끊는 사고를 방지.
- **사용 신호 탐지(휴리스틱)** — `ip xfrm`, IKE 포트, strongswan/libreswan, AFS 마운트, 컨테이너 런타임 확인.
- **결론 중심 출력, 기본 오프라인** — 기본은 진단·조치 요약만 20줄 내외, 근거는 `--verbose`, 네트워크는 `--online`으로 분리.
- **멱등 + 원복 + 자동화** — 재실행해도 결과 동일, `--rollback`으로 원복(sysctl 원래 값 복원), 종료 코드로 Ansible/CI 연동.

## 상세

### 옵션

| 옵션 | 설명 |
|---|---|
| `-n`, `--dry-run` | 점검만 하고 변경 없음(프롬프트 없음) |
| `-y`, `--yes` | 확인 프롬프트 자동 승인(Ansible/CI용). 영향이 큰 네임스페이스 제한은 이 옵션으로 자동 적용되지 않음 |
| `--rollback` | 이 스크립트가 만든 설정을 원복 |
| `-v`, `--verbose` | 판정 근거와 중간 과정을 모두 출력(기본은 요약만) |
| `--online` | 네트워크 사용 허용(기본: 오프라인). 레드햇 계열은 `dnf updateinfo`, 데비안 계열은 `apt changelog` 폴백 |
| `-h`, `--help` / `-V`, `--version` | 도움말 / 버전 |

### 종료 코드

| 코드 | 의미 |
|---|---|
| `0` | 조치 불필요 (또는 원복 완료) |
| `1` | 오류 (root 아님, bash 4 미만 등) |
| `2` | 잘못된 인자 |
| `10` | 조치 필요 (`--dry-run` 결과이거나 사용자 취소) |
| `20` | 변경을 실제로 적용함 |

Ansible에서는 `--yes`와 함께 `failed_when: rc not in [0, 20]` 식으로 연결.

### 배포판 "해당없음" 판정

스크립트가 조치를 건너뛰는 판정은 이 하나뿐이며, 근거는 Red Hat 보안 데이터 API(1차 출처)다.

| CVE / 대상 | 근거 |
|---|---|
| CVE-2026-43500 (RxRPC) / 레드햇 계열 전체 | `package_state` 전 제품 `Not affected`, `affected_release` 없음. 사유: **rxrpc 모듈의 바이너리 RPM을 Red Hat이 출하·지원하지 않음** |

그 외의 "미해당" 주장은 2차 출처에 사실과 다른 내용이 섞여 있어 조치 생략 근거로 쓰지 않는다. 상세는 가이드 4.4절.

### 실행 흐름

1. **사전 점검** — 모듈 로드 상태, 빌트인 여부, 배포판 패치 상태, VPN/컨테이너 사용 신호를 변경 없이 표시.
2. **확인 프롬프트** — `algif_aead`(Copy Fail, KEV 등재 최우선 건)가 조치 대상이면 **독립적으로 먼저** 묻고, 나머지 모듈은 그룹으로 한 번 더 묻는다(`y/N`). 한쪽만 승인해도 승인된 것만 적용되므로, 다른 모듈이 망설여진다는 이유로 가장 시급하고 위험도 낮은 Copy Fail 조치까지 함께 무산되지 않는다. 둘 다 `N`/엔터면 변경 없이 종료.
3. **차단 설정 생성 및 언로드** — 확인 시 `/etc/modprobe.d/mitigate-cve-2026.conf` 생성 후 자동 로드 차단, 로드된 모듈 언로드.
4. **사후 검증** — 로드 상태와 차단 규칙 적용 여부 재확인.
5. **(선택) 빌트인 대응** — 빌트인 코드 감지 시 비특권 user/net 네임스페이스 생성 제한 여부를 추가로 질문(컨테이너/샌드박스 영향 가능).

실행 전 사전 조건, 운영 절차, 롤백 방법은 [pagecache-lpe-guide.ko.md](pagecache-lpe-guide.ko.md) 5절 참고.

### 요구 사항

- **Bash 4+** (연관 배열 사용 — 미만이면 명확한 메시지와 함께 중단)
- **root 권한** (비root 실행 거부)
- 대화형 터미널 (단 `--dry-run`/`--yes` 사용 시 불필요)
- 레드햇 계열: `rpm` / 데비안·우분투 계열: `dpkg`, (선택) `apt`, `timeout`
- `kmod`(modprobe/lsmod), `procps`(sysctl) 없어도 동작 — `/etc/modprobe.d`·`/proc/modules`·`/proc/sys` 직접 읽는 폴백 존재. 단 `modprobe` 없으면 이미 로드된 모듈 언로드는 불가(스크립트가 경고).

### 완화 범위와 한계

| 조치 | 대응 CVE |
|---|---|
| `esp4`/`esp6` 차단 | 43284, 46300, 43503 |
| `rxrpc` 차단 | 43500 |
| `algif_aead` 차단 | 31431 |
| `act_pedit` 차단 | 46331 |
| 비특권 네임스페이스 제한(선택) | 43284/46300/43503/46331 — 커널 코어·COW 결함 대응 |

DirtyClone(43503)의 근본 결함은 커널 코어 함수(`__pskb_copy_fclone()`)에 있어 모듈 차단만으로는 완전히 해결되지 않을 수 있다. 유일한 완전한 해결책은 DirtyFrag+Fragnesia 체인이 모두 포함된 커널(업스트림 v7.1-rc5 이상)로 패치하는 것. 상세는 [pagecache-lpe-guide.ko.md](pagecache-lpe-guide.ko.md) 4절, 5.5절 참고.

> **rxrpc 차단 범위**: 스크립트는 rxrpc의 조치 필요 여부를 43500만으로 판정한다. JFrog 권고의 rxrpc 포함은 계열 전체에 대한 예방적 조치이지 43503의 트리거 경로여서가 아니므로, 43500이 미해당인 환경(RHEL 등)에서 건너뛰어도 43503 경로가 열린 채 남지 않는다. 계열 전체를 예방적으로 굳히려면 수동 차단할 수 있으나 AFS 등 RxRPC 의존 워크로드가 끊긴다.

> **act_pedit 차단**: pedit COW(46331)를 지원한다. `esp4`/`esp6`/`rxrpc`/`algif_aead`와 동일한 방식(자동로드 차단 + 언로드)이며, tc pedit 사용 신호도 함께 탐지한다(휴리스틱). 2순위 그룹과 트리거 전제조건이 같아 비특권 네임스페이스 제한을 적용하면 함께 막힌다.

> **스크립트가 아직 다루지 않는 2건, 제외 사유는 서로 다르다**: PinTheft(43494)는 실사용이 드물어서가 아니라 **자동 차단의 위험이 비대칭적으로 커서** 제외했다 — RDS는 Oracle RAC DB 클러스터 인터커넥트 프로토콜이라 신뢰할 수 있는 사용신호 탐지 방법을 먼저 설계해야 한다. FUSE 캐시 오버플로(31694)는 반대로 **`fuse` 모듈의 실사용 범위가 이 스크립트의 "차단 기본값" 전제보다 넓어서**(sshfs·rclone·Snap 등 의존) 제외했다. 수동 완화 명령과 상세 사유는 [pagecache-lpe-guide.ko.md](pagecache-lpe-guide.ko.md) 5.6절 참고.
